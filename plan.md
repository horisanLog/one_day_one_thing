**ビジネスプランニングドキュメント**

## 1. ビジョン・目的の明文化

### 1.1 ミッションステートメント

**Choose less. Discover more.（選ぶを減らす。発見を増やす。）**

One Day One Thingは、AIが毎朝たった1つの「小さな体験」を提案することで、選択疲れに悩む現代人の日常に偶然の発見（セレンディピティ）を取り戻すWebアプリケーションです。

### 1.2 なぜこのアプリを作るのか

- 選択肢が多すぎて「何をすればいいか分からない」という現代人の疲弊を解消する
- 「遠くに行く」のではなく「今いる場所の豊かさに気づく」という新しいライフスタイルを提案する
- 副業として月10万円の安定収益を実現する持続可能なビジネスを構築する

### 1.3 事業目標

| タイムライン | 目標 |
|:------------|:-----|
| 3ヶ月後 | ベータ版リリース、無料ユーザー50人獲得 |
| 6ヶ月後 | 有料プラン開始（50人達成後）、有料ユーザー100人 |
| 12ヶ月後 | 月収益10万円、有料ユーザー200人以上 |
| 24ヶ月後 | 月収益30万円、ネイティブアプリ化検討 |

> **フェーズ戦略**：最初の50人までは完全無料で提供し、プロダクトの価値検証とフィードバック収集に集中。50人達成後に有料プランを導入する。

## 2. 市場・競合調査

### 2.1 市場概要

One Day One Thingが位置する市場は、「マインドフルネス／ウェルビーイング市場」と「ライフスタイル提案市場」の交差点です。

- **グローバルのウェルビーイングアプリ市場**：2025年時点で約70億ドル規模、年成長率15%以上
- **日本の都市部25〜40歳の単身／DINKS層**：約800万人
- **「休日の過ごし方に困る」層（推定30%）**：約240万人

### 2.2 競合ランドスケープ

| サービス | 概要 | 強み | 当アプリとの違い |
|:---------|:-----|:-----|:----------------|
| アソビュー！ | レジャー体験予約 | 豊富な在庫、割引 | 選択肢が多数→当アプリは1つだけ |
| 習慣化アプリ各種 | ユーザーが設定した習慣を追跡 | 継続率が高い | 自分で設定→当アプリはAIが提案 |
| Headspace / Calm | 瞑想・マインドフルネス | 強いブランド、コンテンツの深さ | 内面の安定→当アプリは外の発見 |
| Shuffle My Life | ランダムに活動を提案 | コンセプトが最も近い | AI未使用、複数提案、個人化なし |

### 2.3 競争優位性

One Day One Thingの競合優位性は「引き算の設計思想」にあります。

- **選択肢を1つに絞る**：既存サービスの真逆のアプローチ
- **日常の中の体験**：遠出や予約不要の「今すぐできる」提案
- **AIパーソナライズ**：天気・気分・履歴から文脈を理解した提案
- **データの蓄積が参入障壁**：使うほどAIが学習し、提案の質が向上

## 4. 課題と提供価値の定義（バリュープロポジション）

### 4.1 コアバリュー

**「Choose less. Discover more.」**— One Day One Thingは選択の負担を取り除き、日常に発見の喜びを取り戻します。

### 4.2 バリュープロポジションキャンバス

**ユーザーのジョブ（達成したいこと）：**
- 休日に意味のあることを見つけたい
- 「今日は良い一日だった」と感じたい
- 大きな努力なしにルーティンから抜け出したい

**ユーザーのペイン（痛み）：**
- 選択肢が多すぎて決定麻痺に陥る
- 「今日も無駄にしてしまった」という罪悪感
- SNSで他人の充実した暮らしと比較してしまう疲労

**ユーザーのゲイン（得られる価値）：**
- 1つの提案が選択不安を解消する
- 毎日の小さな体験が「進んでいる感覚」を生む
- 体験ログの蓄積で「私の日常は思ったより豊かだ」と気づく

### 4.3 既存ソリューションとの主要な違い

| 軸 | 既存アプリ | One Day One Thing |
|:---|:----------|:------------------|
| 選択肢の数 | 多数（10以上） | 1つだけ |
| 対象領域 | レジャー／旅行 | 日常のマイクロ体験 |
| AIの活用 | 検索フィルター程度 | 文脈理解型の生成 |
| 目的 | 効率化／予約 | セレンディピティ／発見 |
| 必要な努力 | 計画して出かける | 今、この場ですぐできる |

## 9. リスク洗い出しと法的確認

### 9.1 主要リスク

| リスク | 深刻度 | 緩和策 | 備考 |
|:-------|:-------|:-------|:-----|
| 提案の質が低い | 高 | 徹底的なプロンプト設計、ユーザーフィードバックループ、A/Bテスト | コアリスク。提案がつまらなければ何もならない |
| 無料で十分と思われる | 中 | 明確なプレミアム価値、段階的な機能制限 | 早期に価格感度をテスト |
| 継続率が低い／解約 | 中 | ストリーク設計、月次レビュー、提案の進化 | 7日/30日継続率を注視 |
| AI APIコストの急騰 | 低 | キャッシュ活用、プロンプト最適化、予算アラート | 現時点では管理可能なコスト |
| 競合がコンセプトを模倣 | 低 | スピード、データの壁、ブランド構築 | アイデアは簡単に真似できるが実行とコミュニティは難しい |
| AIが不適切な提案を生成 | 中 | コンテンツフィルタリング、安全性プロンプト、ユーザー報告機能 | 提案カテゴリの事前レビュー |

### 9.2 法令・コンプライアンス

**対応が必要な法令：**

- **個人情報保護法**：プライバシーポリシー、データ取り扱いの開示、位置情報取得の同意
- **特定商取引法**：オンライン販売における必要記載事項の表示
- **利用規約**：AI生成コンテンツであり専門的アドバイスではない旨の免責事項
- **LINEプラットフォームポリシー**：LINE公式アカウントガイドラインの遵守
- **Stripeコンプライアンス**：PCI DSSはStripe側で対応、適切な実装が必要

**リリース前のアクション：**

- [ ] プライバシーポリシーと利用規約の草案作成（テンプレート活用、弁護士にレビュー依頼）
- [ ] Cookie同意とデータ取り扱いの通知を実装
- [ ] 特定商取引法に基づく表示ページをWebサイトに設置
- [ ] データ削除要求のワークフローを構築（個人情報保護法要件）

## 10. MVPの仮説とスコープの方向性

### 10.1 コア仮説

> 「都市部の25〜40歳に、毎朝LINEでAI生成のマイクロ体験提案を1つ届ければ、30%以上が実行し、5%以上が月額480円の個人化提案に課金する」

### 10.2 MVPスコープ

**作るもの（MVP）：**

- **Webアプリ（レスポンシブ、モバイルファースト）**：LP、サインアップ、今日の提案表示、体験ログ
- **LINE連携**：毎朝の提案配信（LINE公式アカウント経由）
- **AI提案エンジン**：天気・曜日・ユーザー設定を考慮した文脈認識型生成
- **シンプルな記録**：「やった」／「スキップ」 + 任意の1行メモと写真
- **シェアカード**：SNSシェア用の美しいカード自動生成

**作らないもの（MVP後 / 50人達成後）：**

- **Stripe決済**：プレミアムプランのサブスクリプション（50人達成後に導入）
- ネイティブモバイルアプリ
- コミュニティ／ソーシャル機能
- 詳細な分析ダッシュボード
- 細かい位置情報に基づく提案
- シンプルなストリーク以上のゲーミフィケーション

### 10.3 MVP検証基準

50人のベータテスト開始から30日後に、以下を満たせばMVPは検証成功と判断します：

- [ ] 毎日の提案の30%以上が受け入れられている（「やった」）
- [ ] 7日継続率が40%を超えている
- [ ] 少なくとも5人のユーザーが自発的に体験カードをSNSでシェア
- [ ] 定性フィードバックで「毎朝の提案が楽しみ」という声が確認できる
- [ ] 「月額480円払いますか？」に対し、少なくとも20%が「はい」と回答

## 11. コンテンツ戦略（AI提案設計）

> このセクションは本ドキュメントで最も重要な部分です。提案の質がプロダクトそのものです。

### 11.1 提案カテゴリ

| カテゴリ | 説明 | 例 |
|:---------|:-----|:---|
| 五感 | 日常の中で五感を働かす瞬間 | 「目を閉じて、雨の音を5分間聴いてみて」 |
| 探検 | 見慣れた環境の中で何かを発見する | 「今日はいつもと違う改札から出てみて」 |
| プチ冒険 | ルーティンからの小さな逸脱 | 「コンビニで買ったことのない飲み物を1本」 |
| マインドフルネス | 意図的に何もしない、またはペースを落とす | 「ベンチに座って、10分間人を眺めてみて」 |
| つながり | 小さな人とのつながり | 「普段言わない誰かに『ありがとう』を伝えてみて」 |
| 創作 | 何か小さなものを作る | 「気になる影を写真に撮ってみて」 |
| 内省 | 自分の内側を見つめる | 「今週笑顔になったこと3つを書き出してみて」 |

### 11.2 提案の品質基準

- 30分以内、理想的には10分以内でできること
- ゼロコストまたはほぼ無料（「○○を買って」「○○を予約して」は禁止）
- 場所を問わない（自宅でも会社でも外でも可能）
- 説教的でない（「あなたは○○すべき」は禁止）
- 詩的だが気取り過ぎない、温かいが甘過ぎないトーン
- 各提案に「なぜ」を含める（義務ではなく好奇心を喚起する形で）

### 11.3 AI生成に使用するコンテキスト変数

| 変数 | 取得元 | 提案への影響 | 例 |
|:-----|:-------|:-------------|:---|
| 天気 | OpenWeather API | 屋外/屋内の判断 | 雨→五感系／屋内系 |
| 気温 | OpenWeather API | 活動強度の調整 | 寒い→温かい飲み物、心地よい提案 |
| 曜日 | システム時計 | 自由時間の多寡 | 平日→マイクロ、休日→探検系 |
| 季節・暦 | システム + 日本の暦 | 季節感のある提案 | 桜の季節→上を見上げて |
| ユーザーの気分 | 1タップ選択（任意） | 提案のトーン調整 | 疲れ→優しく穏やかな提案 |
| 履歴 | DB（過去の受入／スキップ） | 重複回避、嗜好学習 | 食系が好き→食系を多めに |
| エリアタイプ | ユーザープロフィール | 実現可能性 | 都市部→カフェ、郊外→自然 |

### 11.4 AIプロンプト設計の方向性

AIプロンプトは以下の構造で設計します：

1. **システムプロンプト**：ペルソナ（「優しく詩的なガイド」）、品質ルール、カテゴリバランスを定義
2. **ユーザーコンテキスト**：変数（天気、気分、履歴等）を注入
3. **出力フォーマット**：タイトル（1行）、本文（2〜3行）、結び（1行の詩的な一文）
4. **安全フィルター**：金銭的負担、危険な活動、社会的プレッシャーを含む提案を禁止

> 詳細なプロンプトエンジニアリングは開発フェーズで広範なテストを行いながら実施します。

## 12. ブランド・世界観の定義

### 12.1 ブランドアイデンティティ

| 要素 | 定義 |
|:-----|:-----|
| ブランド名 | One Day One Thing |
| タグライン | Choose less. Discover more.（選ぶを減らす。発見を増やす。） |
| トーン・オブ・ボイス | 温かく、優しく、詩的だが地に足がついている。コーチが命令するのではなく、親しい友人がそっと囁くような感覚。 |
| ビジュアルスタイル | ミニマリスト、落ち着いたアーストーン、十分な余白。日本の美意識（侘寂）にインスパイア。 |
| タイポグラフィ | UIにはクリーンなゴシック体、提案テキストには柔らかな明朝体または丸ゴシック |
| カラーパレット | 温かい白 (#FAFAF7)、柔らかな緑 (#7C9A82)、淡い青 (#8BA4B8)、温かいグレー (#9B9B8E) |
| 「私たちではないもの」 | 生産性ツールではない。ToDoリストではない。旅行ガイドではない。自己啓発アプリではない。 |

### 12.2 ボイス＆トーンガイドライン

提案テキストが目指す感覚：

> 「あなたが毎日通り過ぎていた美しいものに気づく、優しい友人の囁き」

**やるべきこと：**

- 五感に訴える言葉を使う（「温もり」「音」「感じてみて」）
- 具体的だが限定しない（「買ったことのない飲み物」→「コカ・コーラゼロを買って」ではない）
- ユーザーの解釈の余地を残す
- 天気、季節、時間帯を自然に反映する

**やってはいけないこと：**

- 命令形を使う（「○○しなさい」「今すぐ○○して」）
- 生産性用語を使う（「最適化」「レベルアップ」「達成」）
- 説教する（「これはあなたのためになるから…」）
- ブランド、商品、特定の場所を名指しする

### 12.3 コンテキスト別の提案例

**平日の朝、曇り、気分「普通」：**

> 「通勤途中、今日の空のグレーの濃淡を数えてみて。思ったよりたくさんの色が見つかるかもしれません」

**休日の午後、晴れ、気分「良い」：**

> 「家のどこかにある日だまりを探してみて。その中に数分座って、肌を伝う温もりを感じてみてください」

**平日の夕方、雨、気分「疲れた」：**

> 「今夜は、何もしない。本当に何も。スクリーンも、本も、音楽もなし。ただ座って、雨の音だけで十分です」

## 付録：アーキテクチャ概要

Webからネイティブアプリへのシームレスな移行を実現するシステムアーキテクチャです。

すべてのユーザーデータ、認証、決済状態はサーバー側のAPIで一元管理され、将来のクライアント（ネイティブアプリ、タブレット等）は同じバックエンドに接続し、アカウントとサブスクリプションの引き継ぎを保証します。

| コンポーネント | 現在（MVP） | 将来（スケール時） |
|:--------------|:-----------|:------------------|
| ユーザーインターフェース | Next.js Webアプリ（レスポンシブ） | + ネイティブアプリ（iOS/Android） |
| 通知 | LINE公式アカウント | + プッシュ通知（ネイティブ） |
| 認証 | Supabase Auth（Google + メール） | 同一（Apple Sign-in追加） |
| 決済 | Stripe（※50人達成後に導入） | 同一（ネイティブからWeb決済） |
| AI | Claude API | + 必要に応じてファインチューンモデル |
| データベース | Supabase（PostgreSQL） | 同一 または専用DBへ移行 |
